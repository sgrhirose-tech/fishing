## 1. 概要・目的

### 背景

Tsuricastは神奈川・千葉・東京などの釣り場（205スポット）に対して天気・波高などの海況情報を提供するサービスである。現状は気象・波浪データを Open-Meteo Marine API から取得しているが、潮汐情報（満潮・干潮の時刻と高さ、潮名）が未実装である。

潮汐は釣り人にとって最重要データのひとつであり、特にシロギスをはじめとした砂浜の魚は潮の動きと釣果が強く連動する。

### 目的

tide736.net が提供する潮汐APIを活用し、各釣り場ページに以下の情報を追加することで、Tsuricastの付加価値を高める。

- 本日の満潮・干潮の時刻と潮位（cm）
- 潮名（大潮 / 中潮 / 小潮 / 長潮 / 若潮）
- タイドグラフ（1日分の潮位曲線）

### 設計方針の核心

> tide736.netは**個人運営の無料サービス**であり、アクセス上限数は非公開・予告なくサービス終了の可能性がある。  
> そのため、**取得頻度を最小化し、自前でデータをキャッシュ保持する**ことを最重要方針とする。

## 2. tide736.net APIの仕様

### 2-1. エンドポイント

```
GET https://api.tide736.net/get_tide.php
```

認証キーは不要。パラメータをクエリ文字列で渡すだけで使用できる。

### 2-2. リクエストパラメータ

|パラメータ|意味     |値の例                          |必須|
|-----|-------|-----------------------------|--|
|`pc` |都道府県コード|`14`（神奈川）, `12`（千葉）, `13`（東京）|✅ |
|`hc` |港コード   |`5`（例：小田原）                   |✅ |
|`yr` |年      |`2026`                       |✅ |
|`mn` |月      |`4`                          |✅ |
|`dy` |日      |`2`                          |✅ |
|`rg` |取得範囲   |`day` / `week` / `month`     |✅ |

**重要**: `rg=month` を使うと、指定した日を含む月の全データを**1リクエストで**取得できる。アクセス回数削減のため、原則 `rg=month` を使用する。

#### リクエスト例（神奈川・小田原港・2026年4月分を一括取得）

```
https://api.tide736.net/get_tide.php?pc=14&hc=5&yr=2026&mn=4&dy=1&rg=month
```

### 2-3. データの特性（重要）

潮汐データは**天体の動きから数学的に計算した推算値**であり、気象予報とは異なる。

|項目       |気象（天気・波）|潮汐         |
|---------|--------|-----------|
|データ種別    |観測・予測   |計算値        |
|何日先まで取得可能|7〜16日程度 |**何年先でも可能**|
|更新の必要性   |毎時〜数時間ごと|**月1回で十分** |

→ このため、潮汐データは月初に1ヶ月分まとめて取得し、以降は保存済みデータを使い回す運用が最適。

### 2-4. 画像API（タイドグラフ画像）

グラフ画像を直接返すエンドポイントも存在する。

```
GET https://api.tide736.net/tide_image.php
```

|パラメータ                       |説明       |例            |
|----------------------------|---------|-------------|
|`pc`, `hc`, `yr`, `mn`, `dy`|上記と同じ    |`pc=14&hc=5` |
|`rg`                        |表示範囲     |`day`（1日分）   |
|`w`, `h`                    |画像サイズ（px）|`w=800&h=400`|
|`lc`                        |折れ線の色    |`blue`       |
|`ld`                        |凡例表示     |`on` / `off` |
|`ttd`                       |満干潮マーカー表示|`on` / `off` |

**利用方針**: 画像APIはフロントエンドの実装コストを大幅に下げられるが、tide736.net のサーバーに毎ページビューでリクエストが飛ぶ。サービス継続性リスクの観点から、本設計では**JSONデータをキャッシュし自前でグラフを描画する**ことを推奨する。画像APIは開発初期のプロトタイプ段階のみ使用可。

## 3. データフロー設計

```
[tide736.net API]
        │
        │ 月1回 / 港ごと（rg=month）
        ▼
[FastAPI バッチ処理]
        │
        │ 正規化・変換してDB保存
        ▼
[PostgreSQL / SQLite]
  tidal_cache テーブル
        │
        │ ユーザーアクセス時
        ▼
[FastAPI エンドポイント]
  GET /api/spots/{spot_id}/tide?date=YYYY-MM-DD
        │
        ▼
[Cloudflare Pages フロントエンド]
  タイドグラフ・潮汐サマリーを表示
```

### ポイント

- tide736.netへの直接リクエストは**バッチ処理のみ**が行う。フロントエンドから tide736.net を直接叩くことは禁止。
- ユーザーからのリクエストは常にTsuricastのAPIサーバーが応答する。
- これにより、tide736.netのサービス停止や変更の影響をユーザーから隔離できる。

## 4. キャッシュ戦略

### 4-1. 保存対象データ

1ヶ月分のJSONを取得したあと、DBには以下の単位でレコードを保存する。

```
港コード（harbor_code）× 日付（date） = 1レコード
```

1レコードに含む内容：

- 満潮・干潮の時刻と潮位（最大4件/日）
- 潮名（大潮・中潮など）
- 1時間ごとの潮位リスト（24点）
- 日の出・日の入り時刻

### 4-2. キャッシュ有効期間

|データ種別   |有効期間     |理由         |
|--------|---------|-----------|
|当月の潮汐データ|月末まで有効   |計算値のため変わらない|
|翌月以降のデータ|永続（上書き不要）|同上         |

**更新が必要になるケース**: tide736.net 側で計算ロジックが変更されたとき（極めて稀）。その場合はバッチを手動で再実行して上書きする。

### 4-3. 港と釣り場スポットのマッピング

Tsuricastの釣り場スポットと、tide736.netの港コードは1対1ではない。

```
釣り場スポット（205か所）
    │
    │ 最寄り港を1つ割り当て
    ▼
tide736.net 港コード（神奈川・千葉・東京内の港）
```

マッピングはマスタJSONファイルで管理する（詳細は[7章](#7-港コードマスタの管理方針)参照）。

## 5. 取得スケジュール設計

### 5-1. 通常運用

```
毎月1日 AM 3:00（JST）
    │
    ├─ 対象港コードのリストを読み込む
    │
    ├─ 各港に対して rg=month でAPIリクエスト
    │     └─ 1秒待機（連続アクセス防止）
    │
    ├─ 取得データをDBに保存
    │
    └─ 完了ログを記録
```

### 5-2. 取得リクエスト数の見積もり

対象エリアの港数を仮に80港とすると：

|項目          |数値  |
|------------|----|
|港数          |80港 |
|月あたりリクエスト数  |80回 |
|年間リクエスト数    |960回|
|1リクエストあたりの待機|1秒  |
|バッチ所要時間     |約80秒|

これは非常に小さいアクセス量であり、制限に引っかかる心配は実質ない。

### 5-3. 翌月分の先行取得（推奨）

月末に翌月分を先行取得しておくと、月初のバッチ失敗時のリスクを下げられる。

```
毎月25日 AM 3:00
    │
    └─ 翌月分を同様の手順で取得・保存
```

### 5-4. バッチ失敗時のフォールバック

取得に失敗した港があった場合：

1. エラーログに港コードと日付を記録する
1. 翌日AM3:00に失敗港のみ再試行する
1. それでも失敗した場合は、直近の保存済みデータをそのまま使用する（潮汐は変わらないため品質上の問題は少ない）

## 6. JSONデータ構造とフィールド定義

### 6-1. tide736.net APIのレスポンス構造

APIは以下の形式でJSONを返す（`rg=day` で1日分を取得した場合の例）。

```json
{
  "tide": {
    "chart": {
      "2026-04-02": {
        "sun": {
          "rise": "05:30",
          "set":  "18:10",
          "astro_twilight":   ["04:02", "19:38"],
          "regular_twilight": ["05:05", "19:05"]
        },
        "moon": {
          "title":      "中潮",
          "age":        "10.9",
          "brightness": "79.1",
          "rise":       "1日 14:38",
          "set":        "2日 03:34"
        },
        "flood": [
          { "time": "03:43", "unix": 1743518580000, "cm": 172.9 },
          { "time": "15:41", "unix": 1743561660000, "cm": 168.2 }
        ],
        "ebb": [
          { "time": "09:48", "unix": 1743540480000, "cm": 78.2 },
          { "time": "21:55", "unix": 1743583500000, "cm": 54.8 }
        ],
        "tide": [
          { "time": "00:00", "unix": 1743519600000, "cm": 145.0 },
          { "time": "01:00", "unix": 1743523200000, "cm": 162.5 }
          // 以降1時間ごとに24件
        ]
      }
    }
  }
}
```

### 6-2. フィールド定義

#### `sun`（太陽情報）

|フィールド             |型               |説明          |
|------------------|----------------|------------|
|`rise`            |string `"HH:MM"`|日の出時刻       |
|`set`             |string `"HH:MM"`|日の入り時刻      |
|`astro_twilight`  |array[2]        |天文薄明の開始・終了時刻|
|`regular_twilight`|array[2]        |常用薄明の開始・終了時刻|

#### `moon`（月・潮名情報）

|フィールド       |型         |説明                                           |
|------------|----------|---------------------------------------------|
|`title`     |string    |**潮名**。`大潮` / `中潮` / `小潮` / `長潮` / `若潮` のいずれか|
|`age`       |string（数値）|月齢（0〜29.5）                                   |
|`brightness`|string（数値）|月の輝面比（%）                                     |
|`rise`      |string    |月の出時刻（「1日 14:38」形式）                          |
|`set`       |string    |月の入り時刻                                       |

#### `flood`（満潮リスト）

1日に最大2件。配列の要素数が1件の日もある。

|フィールド |型               |説明         |
|------|----------------|-----------|
|`time`|string `"HH:MM"`|満潮時刻       |
|`unix`|number          |UNIX時刻（ミリ秒）|
|`cm`  |number          |潮位（cm）     |

#### `ebb`（干潮リスト）

1日に最大2件。`flood` と同じ構造。

#### `tide`（時刻別潮位リスト）

1時間ごと、1日24件。タイドグラフ描画に使用する。

|フィールド |型               |説明             |
|------|----------------|---------------|
|`time`|string `"HH:MM"`|時刻（00:00〜23:00）|
|`unix`|number          |UNIX時刻（ミリ秒）    |
|`cm`  |number          |潮位（cm）         |

### 6-3. Tsuricast内部の正規化スキーマ

取得したJSONをDBに保存する際のテーブル定義。

```sql
CREATE TABLE tidal_cache (
    id            SERIAL PRIMARY KEY,
    harbor_code   VARCHAR(10)  NOT NULL,  -- "14-5" のような "pc-hc" 形式
    date          DATE         NOT NULL,
    tide_name     VARCHAR(10)  NOT NULL,  -- 潮名
    sunrise       TIME,
    sunset        TIME,
    moon_age      NUMERIC(4,1),
    flood_json    JSONB,                  -- 満潮リスト（最大2件）
    ebb_json      JSONB,                  -- 干潮リスト（最大2件）
    hourly_json   JSONB,                  -- 時刻別潮位（24件）
    fetched_at    TIMESTAMP    NOT NULL DEFAULT NOW(),
    UNIQUE (harbor_code, date)
);
```

**`harbor_code` の形式について**  
tide736.netの都道府県コード（`pc`）と港コード（`hc`）を `pc-hc` 形式で結合して管理する。例：神奈川（14）・小田原港（5）→ `"14-5"`

## 7. 港コードマスタの管理方針

### 7-1. マスタJSONの構造

釣り場スポットと港コードのマッピングをJSONファイルで管理する。

```json
{
  "spots": {
    "kanagawa-odawara-fishing-park": {
      "harbor_code": "14-5",
      "harbor_name": "小田原",
      "note": "小田原漁港が最寄り港"
    },
    "chiba-katsuura-beach": {
      "harbor_code": "12-8",
      "harbor_name": "勝浦",
      "note": null
    }
  }
}
```

ファイル名: `harbor_mapping.json`  
管理場所: バックエンドリポジトリの `data/` ディレクトリ

### 7-2. 最寄り港の選び方

釣り場スポットに対して最寄りの港を1つ割り当てる。選定基準は以下の順：

1. スポットと同じ市町村内の港
1. 直線距離が最も近い港
1. 湾の形状的に潮汐特性が近い港（湾内 vs 外洋）

> 潮汐の時刻は数km離れた港でも数分程度しかずれないため、完全一致でなくて問題ない。

### 7-3. 更新タイミング

新規スポットを追加するたびに `harbor_mapping.json` に行を追加し、**次月のバッチ実行時に自動的に取得対象に含まれる**設計とする。

## 8. エラーハンドリング

### 8-1. APIリクエスト失敗時

|状況              |対応               |
|----------------|-----------------|
|タイムアウト（10秒）     |3回リトライ後にスキップ・ログ記録|
|HTTP 5xx        |同上               |
|レスポンスが空 / 不正JSON|ログ記録・スキップ        |
|rate limit（429等）|待機時間を30秒に延ばして再試行 |

### 8-2. フロントエンドへの影響抑制

DBにデータが存在しないスポット・日付に対してAPIリクエストが来た場合：

```json
{
  "tide": null,
  "tide_unavailable_reason": "data_not_fetched"
}
```

フロントエンドは `tide` が `null` の場合、潮汐エリアを非表示にする（エラー表示はしない）。

### 8-3. tide736.netのサービス停止時

保存済みのキャッシュデータがある限り、ユーザーへの影響はゼロ。潮汐は計算値のため、数年分のデータを事前にまとめ取りしておくことも技術的には可能。

## 9. フロントエンドへの提供インターフェース

### 9-1. エンドポイント定義

```
GET /api/spots/{spot_id}/tide?date=YYYY-MM-DD
```

|パラメータ    |説明                                              |
|---------|------------------------------------------------|
|`spot_id`|釣り場スポットのslug（例: `kanagawa-odawara-fishing-park`）|
|`date`   |表示したい日付（省略時は当日）                                 |

### 9-2. レスポンスJSON

```json
{
  "date": "2026-04-02",
  "harbor_name": "小田原",
  "tide_name": "中潮",
  "sunrise": "05:30",
  "sunset": "18:10",
  "moon_age": 10.9,
  "flood": [
    { "time": "03:43", "cm": 172.9 },
    { "time": "15:41", "cm": 168.2 }
  ],
  "ebb": [
    { "time": "09:48", "cm": 78.2 },
    { "time": "21:55", "cm": 54.8 }
  ],
  "hourly": [
    { "time": "00:00", "cm": 145.0 },
    { "time": "01:00", "cm": 162.5 },
    // ... 24件
    { "time": "23:00", "cm": 130.0 }
  ]
}
```

### 9-3. タイドグラフの描画仕様

フロントエンドは `hourly` の24点のデータを使い、折れ線グラフを描画する。

- X軸: 時刻（00:00〜23:00）
- Y軸: 潮位（cm）、範囲は0〜最大潮位の1.2倍程度
- `flood` の時刻に「▲ 満潮」マーカーを表示
- `ebb` の時刻に「▽ 干潮」マーカーを表示
- グラフライブラリ: Chart.js または Recharts を推奨

## 10. 運用上の注意事項

### 絶対にやってはいけないこと

|禁止事項                       |理由                                         |
|---------------------------|-------------------------------------------|
|フロントエンドから tide736.net を直接叩く|ユーザー数 × アクセス数が tide736.net に直撃し、制限・停止の原因になる|
|1日複数回の再取得                  |計算値のため更新の必要がなく、無意味なアクセスになる                 |
|バッチを手動で何度も実行する             |同上                                         |

### 免責事項の表示

tide736.netのデータは推算値であり、航海用途には使用できない。釣り場ページの潮汐情報の近くに以下の文言を表示すること。

> 潮汐データは tide736.net の推算値を使用しています。実際の潮位と異なる場合があります。航海には海上保安庁発行の潮汐表をご使用ください。

### ビジネスプランの検討

tide736.net は「専用サーバー（ビジネスプラン）」の提供もアナウンスしている。Tsuricastが本格的にスケールした場合や、より安定した稼働が必要になった場合は、ビジネスプランへの移行を検討すること（`tide736@gmail.com` 宛に連絡）。

## 付録A. Tsuricast 実装メモ（設計書との差分）

**更新日**: 2026-04-01

本設計書は PostgreSQL / FastAPI 構成を想定して書かれているが、Tsuricast は
**FastAPI + JSON ファイルキャッシュ + GitHub API プッシュ** の構成のため、以下の変換を行って実装した。

### A-1. DB → JSON ファイル

設計書 §6-3 の `tidal_cache` テーブルに相当するデータを、以下の JSON ファイルとして保存する。

- **ファイルパス**: `data/tides/{harbor_code}_{YYYY-MM}.json`  
  例: `data/tides/14-5_2026-04.json`
- **1ファイル = 1港 × 1ヶ月分**。日付をキーとする `days` オブジェクト。
- **重複排除**: ファイル存在確認でスキップ（SQL の `UNIQUE` 制約に相当）。
- **永続保存**: GitHub リポジトリの `data/tides/` ディレクトリにコミット。

```json
{
  "_meta": {
    "harbor_code": "14-5",
    "harbor_name": "小田原",
    "month": "2026-04",
    "fetched_at": "2026-04-01T03:00:00+09:00"
  },
  "days": {
    "2026-04-01": {
      "tide_name": "中潮",
      "sunrise": "05:30",
      "sunset": "18:10",
      "moon_age": 10.9,
      "flood": [{"time": "03:43", "cm": 172.9}, {"time": "15:41", "cm": 168.2}],
      "ebb":   [{"time": "09:48", "cm": 78.2},  {"time": "21:55", "cm": 54.8}],
      "hourly": [{"time": "00:00", "cm": 145.0}, ...]
    }
  }
}
```

### A-2. 実装ファイル構成

| ファイル | 役割 |
|---|---|
| `data/harbor_mapping.json` | spot slug → tide736.net 港コード マッピング（§7 対応）|
| `data/tides/` | 月別潮汐 JSON キャッシュの置き場 |
| `scripts/fetch_tides.py` | 月次バッチスクリプト（§5 対応）|
| `app/tides.py` | 潮汐データ読み込みモジュール |

### A-3. Render cron スケジュール

render.yaml に2本のジョブを追加。

| ジョブ名 | schedule (UTC) | 実行内容 |
|---|---|---|
| `tsuricast-tides-monthly` | `0 18 1 * *`（毎月2日 AM3 JST）| 当月＋翌月を取得 |
| `tsuricast-tides-prefetch` | `0 18 24 * *`（毎月25日 AM3 JST）| 翌月のみ先行取得 |

### A-4. API エンドポイント

設計書 §9 のエンドポイントを FastAPI で実装。  
`app/main.py` に追加済み。

```
GET /api/spots/{slug}/tide?date=YYYY-MM-DD
```

- `slug`: spots/ ディレクトリのファイル名（スポット識別子）
- `date`: 省略時は当日（JST）
- `data/tides/` にキャッシュがない場合: `{"tide": null, "tide_unavailable_reason": "data_not_fetched"}`

### A-5. harbor_mapping.json の追記方法

スポットに潮汐データを紐づけるには、`data/harbor_mapping.json` の `spots` オブジェクトに追記する。

```json
{
  "spots": {
    "spot-slug-here": {
      "harbor_code": "14-5",
      "harbor_name": "小田原",
      "note": "小田原漁港が最寄り港"
    }
  }
}
```

追記後、次月のバッチ実行時に自動的に取得対象に含まれる。  
即時取得が必要な場合は以下のコマンドを実行:

```bash
GITHUB_TOKEN=xxx python scripts/fetch_tides.py --harbor 14-5
```
