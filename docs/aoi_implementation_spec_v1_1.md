**バージョン**: v1.1
**作成日**: 2026-04-27
**対象実装者**: Claude Code

v1.1の更新点:
- イラスト配置先を `/static/img/aoi/` に変更
- プロンプトファイル名を `aoi_prompt.md` に変更
- 釣り禁止スポットには実装しないルールを追加

## 1. 概要

### 1.1 目的

Tsuricast の各釣りスポットページに、AIキャラクター「中村葵」(以下、葵ちゃん)が今日と明日のコンディション判定コメントを表示する機能を実装する。

### 1.2 背景

これまで16スポット限定でテスト運用してきた葵ちゃんコメント生成機能を、サイト全体(約650スポット)に展開する。コスト最適化のため、**全スポット一斉生成方式ではなくアクセス連動生成方式**を採用する。

### 1.3 ゴール

- 各スポットページにアクセス時、葵ちゃんコメント(今日・明日)が表示される
- アクセスのあったスポットだけコメントを生成し、一定期間キャッシュで再利用する
- ピーク日(約100スポット/日アクセス)でも月額予算内で運用できる
- プロンプトキャッシングを活用し、コストを最小化する

## 2. 機能要件

### 2.1 コメント生成タイミング

**アクセス連動生成 + キャッシュ方式**

- ユーザーがスポットページにアクセスしたとき、対象スポットの今日/明日のコメントがキャッシュにあるか確認する
- キャッシュにあれば即座に表示する
- キャッシュになければ、その場で生成してキャッシュに保存し、表示する
- 検索エンジンクローラーや bot からのアクセスでは生成しない(後述「クローラー対策」参照)

**釣り禁止スポットには実装しない**

スポットマスターで「釣り禁止」フラグが立っているスポットには葵ちゃんコメントを表示しない。
コメント生成のAPI呼び出しも行わない。
釣り禁止スポットでコメントを出すと「ここで釣りをしていい」という誤解を招くため、
釣り禁止判定はコメント表示判定の最優先条件とする。

判定フロー:
```
1. スポットが釣り禁止 → コメント非表示(API呼び出しなし、終了)
2. アクセス元がbot → コメント非表示(API呼び出しなし、終了)
3. キャッシュにヒット → キャッシュからコメント表示
4. キャッシュミス → API呼び出し → 生成 → キャッシュ保存 → 表示
```

### 2.2 生成単位

1スポットにつき2コメント(今日用・明日用)を独立に生成する。
これは既存のテスト運用と同じ仕様。

### 2.3 キャッシュの仕組み

#### 2.3.1 アプリケーション側キャッシュ(コメント本体のキャッシュ)

各スポットの生成コメントを、データベース or KVストアに保存する。

**キャッシュキー**: `{spot_id}:{date_label}:{date}`
- 例: `chigasaki-kaigan:today:2026-04-27`

**キャッシュ期間**:
- 今日コメント: 当日中(深夜0時に失効)
- 明日コメント: 翌日0時に失効(つまり当日中だけ有効)

**キャッシュ更新タイミング**:
- 通常: ページアクセス時に存在確認、なければ生成して保存
- 強制更新: 気象データの大幅変動を検知した場合(後述「気象データ変動時の対応」参照)

#### 2.3.2 Anthropic API側のプロンプトキャッシング

**これは必須要件です。** 実装しないとコストが約7倍になる。

- Anthropic API の `cache_control` 機能を利用
- SYSTEM部(プロンプト本体、約16,500トークン)をキャッシュ対象とする
- USER部(各スポット個別データ、約350トークン)はキャッシュ対象外
- キャッシュ期間: **1時間**を推奨(5分版より割高だが、書き込み頻度が下がりトータルで安い)

**実装例(Python anthropic SDK)**:
```python
response = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=200,
    system=[
        {
            "type": "text",
            "text": SYSTEM_PROMPT,  # プロンプトv3.6の本体
            "cache_control": {"type": "ephemeral"}  # 1時間キャッシュ
        }
    ],
    messages=[
        {
            "role": "user",
            "content": user_message  # スポット個別データ
        }
    ]
)
```

### 2.4 表示要件

- 葵ちゃんコメント本文(80〜120字)
- 葵ちゃんイラスト(モード判定で4種類使い分け、後述「イラスト連動」)
- スポット名と日付ラベル(今日/明日)
- 「AIによる生成です」の旨を控えめに表記(法的・倫理的配慮)

### 2.5 イラスト連動

葵ちゃんの4モードイラスト(good/unsure/ng/danger)をコメントの判定モードに合わせて表示する。

#### 2.5.1 モード判定ロジック

コメント文の冒頭フレーズで判定する:

| モード | 判定キーワード(冒頭) | 表示イラスト |
|---|---|---|
| good | 「これは行く日」「完全に勝ち」「迷う必要ありません」「コンディション出来上がってます」 | weather_good.png |
| unsure | 「ちょっと迷います」「判断微妙」「ここがネック」「行くなら○○前提で」 | weather_unsure.png |
| ng | 「家の日にします」「雨なので家の日」「装備しても厳しい」 | weather_ng.png |
| danger | 「安全第一でやめときます」「無理は禁物」 | weather_danger.png |

判定ロジック実装例:
```python
def detect_mode(comment_text: str) -> str:
    # ステップ1: 危険判定（最優先）
    if "安全第一" in comment_text or "無理は禁物" in comment_text:
        return "danger"
    # ステップ2: 家の日判定
    if "家の日" in comment_text or "装備しても厳しい" in comment_text:
        return "ng"
    # ステップ3: 行く日判定
    if any(phrase in comment_text for phrase in [
        "これは行く日", "完全に勝ち", "迷う必要ありません",
        "コンディション出来上がってます", "ベタ凪ぎ"
    ]):
        return "good"
    # ステップ4: それ以外は悩む日
    return "unsure"
```

**注意**: コメント文がイレギュラーな場合(プロンプトの想定外のフレーズ)は `unsure` をデフォルトとする。

### 2.6 クローラー対策

検索エンジンクローラーや bot からのアクセスでコメント生成しない。

#### 判定方法
1. User-Agent によるbot判定(Googlebot, Bingbot, 各種クローラー)
2. アクセス頻度による判定(同一IPから1秒以内に複数ページアクセス等)
3. ヘッドレスブラウザ判定(Playwrightやpuppeteerなど)

#### bot判定時の挙動
- キャッシュがあれば返す
- キャッシュがなければ、葵ちゃんコメントなしでページを返す(またはプレースホルダー表示)
- API呼び出しは行わない(コスト発生しない)

## 3. 入力データ仕様

### 3.1 必要な入力データ

各スポットの今日/明日のコメント生成に必要なデータ。Open-Meteo API等から取得すること(既存の実装に準ずる)。

| 項目 | データ型 | 例 | 備考 |
|---|---|---|---|
| spot_name | string | "茅ヶ崎海岸" | スポット表示名 |
| date_label | string | "今日" or "明日" | 固定2値 |
| month | int | 4 | 1〜12 |
| weather | string | "快晴" "雨" "大雨" 等 | |
| temp_min | float | 11.4 | ℃ |
| temp_max | float | 18.6 | ℃ |
| wave | float | 0.6 | m |
| wind_dir | string | "北北東" | 16方位 |
| wind_speed | float | 3.7 | m/s |
| period | float | 6.4 | 波周期(秒) |
| sea_temp | float | 18.6 | 海水温(℃) |
| tide_info | string | "若潮(月齢9.6)" | 潮汐情報 |
| precip_morning | int | 0 | 朝の降水量(mm/h) |
| precip_noon | int | 0 | 昼の降水量(mm/h) |
| precip_evening | int | 0 | 夕の降水量(mm/h) |
| precip_night | int | 0 | 夜の降水量(mm/h) |
| spot_type | string | "sand_beach" | 後述参照 |

### 3.2 spot_type の取りうる値

スポットの地形タイプ。既存のスポットマスターから取得する。

| 値 | 意味 |
|---|---|
| sand_beach | 砂浜 |
| breakwater | 堤防 |
| fishing_facility | 釣り施設(港湾、釣り公園等) |
| rocky_shore | 磯 |

これは葵ちゃんプロンプトの「波判定」セクションで地形別の閾値を切り替えるために使う。

### 3.3 合成変数(コード側で前処理が必要)

プロンプト内で `{wind_relative_clause}` 等の変数を使う前に、以下の前処理が必要:

#### `{wind_relative_clause}`
釣り場の正面(facing direction)と風向きから、追い風/向かい風/横風/斜めを判定し、
「(追い風)」「(向かい風)」「(横風)」「(斜め追い風)」等の括弧付き文字列を生成する。
値が決定できない場合は空文字。

実装の擬似コード:
```python
def get_wind_relative(wind_dir: str, facing: str) -> str:
    angle_diff = calc_angle_diff(wind_dir, facing)
    if angle_diff < 22.5:
        return "（向かい風）"
    elif angle_diff < 67.5:
        return "（斜め向かい風）"
    elif angle_diff < 112.5:
        return "（横風）"
    elif angle_diff < 157.5:
        return "（斜め追い風）"
    else:
        return "（追い風）"
```

#### `{tide_activity_clause}`
潮汐の動きが活発か穏やかかを判定し、「(潮の動き:活発)」「(潮の動き:穏やか)」を返す。判定できなければ空文字。

判定ロジック例:
- 大潮、中潮 → 「(潮の動き:活発)」
- 若潮、長潮 → 「(潮の動き:穏やか)」
- 小潮 → 空文字

#### `{facing_line}`
釣り場の正面方位がある場合、改行付きで「\n釣り場の正面:SE」のような文字列を返す。
値がなければ空文字。

### 3.4 USER部のテンプレート

実際にAPIに送るUSER部のテンプレートは以下(プロンプトv3.6の仕様):

```
{date_label}の{spot_name}
月：{month}月
天気：{weather}
最低気温：{temp_min}℃
最高気温：{temp_max}℃
波高：{wave}m
風：{wind_dir} {wind_speed}m/s{wind_relative_clause}
波周期：{period}s
海水温：{sea_temp}℃
潮汐：{tide_info}{tide_activity_clause}
降水量(mm/h)：朝{precip_morning}, 昼{precip_noon}, 夕{precip_evening}, 夜{precip_night}
施設区分：{spot_type}{facing_line}
```

## 4. API呼び出し仕様

### 4.1 使用モデル

```
claude-haiku-4-5-20251001
```

### 4.2 パラメータ

| パラメータ | 値 | 備考 |
|---|---|---|
| max_tokens | 200 | 出力制限 |
| temperature | デフォルト(指定なし) | |

### 4.3 SYSTEM部

ファイル `aoi_prompt.md` (中身はプロンプトv3.6) の SYSTEM セクション全体を渡す。
ファイルは `/opt/render/project/src/aoi_prompt.md` に配置されている。

**重要**: SYSTEM部は **必ずプロンプトキャッシングを有効化**すること。

### 4.4 タイムアウト

- 接続タイムアウト: 10秒
- レスポンスタイムアウト: 30秒
- 失敗時はリトライ1回(指数バックオフ)、それでも失敗したらコメントなしで表示

### 4.5 エラー処理

#### API呼び出し失敗時

**葵ちゃんコメント部分は何も表示しない**(空白)。
プレースホルダーや「準備中です」のメッセージは出さない。
- ユーザーには「コメント機能がない通常のスポットページ」として見える
- ログには記録(後述「日次運用ログ」参照)
- リトライは1回まで(指数バックオフ)、それでも失敗なら諦める

#### レート制限ヒット時
- 指数バックオフでリトライ(最大1回)
- それでも失敗なら何も表示しない

## 5. データベース・キャッシュ要件

### 5.1 ストレージ選択

以下のいずれかを採用(運用環境に合わせて選択):
- **Redis** (推奨、TTL機能が標準装備)
- PostgreSQL/MySQL のキャッシュテーブル
- Render Disk + ファイルベース(小規模なら可)

### 5.2 スキーマ例

```sql
CREATE TABLE aoi_comment_cache (
    cache_key VARCHAR(255) PRIMARY KEY,  -- "{spot_id}:{date_label}:{date}"
    spot_id VARCHAR(100) NOT NULL,
    date_label VARCHAR(10) NOT NULL,     -- "today" or "tomorrow"
    target_date DATE NOT NULL,
    comment_text TEXT NOT NULL,
    mode VARCHAR(20) NOT NULL,            -- "good"/"unsure"/"ng"/"danger"
    created_at TIMESTAMP NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    weather_data_hash VARCHAR(64),        -- 生成元データのハッシュ(後述)
    INDEX idx_spot_date (spot_id, target_date)
);
```

### 5.3 weather_data_hash の用途

生成元の気象データのハッシュを保存する。気象データが大幅変動した場合、キャッシュを破棄して再生成する判断材料にする。

ハッシュ対象:
- 波高(0.1m単位で丸め)
- 風速(1m/s単位で丸め)
- 降水量4時間帯(整数mm/h)
- 天気カテゴリ

```python
def calc_weather_hash(data: dict) -> str:
    key = f"{round(data['wave'], 1)}_{round(data['wind_speed'])}_" \
          f"{data['precip_morning']}_{data['precip_noon']}_" \
          f"{data['precip_evening']}_{data['precip_night']}_" \
          f"{data['weather']}"
    return hashlib.sha256(key.encode()).hexdigest()[:16]
```

### 5.4 気象データ変動時の対応

オプショナル機能。実装の優先度は中。

ページアクセス時、キャッシュにコメントがあっても、現在の気象データのハッシュとキャッシュ内のハッシュが大きく違う場合は再生成する。
ただし、これを毎アクセスでチェックすると意味がないので、**1時間に1回程度**のチェック頻度に絞る。

## 6. コスト管理・運用ログ要件

### 6.1 想定コスト

実測データに基づく試算:
- 1コメント生成: 約2.6円(キャッシュなし) / 約0.4円(プロンプトキャッシュあり)
- ピーク日(約97スポット×2=194コメント): 約505円 / 約129円
- 月額(ピーク基準・キャッシュあり): **約3,900円**

### 6.2 日次運用ログ(メール送信)

**毎日決まった時刻に、運用ログを運営者(admin@tsuricast.jp)宛にメール送信する。**

#### 送信タイミング
- 毎日23時(JST)に当日分のログを集計してメール送信
- または翌日6時(JST)に前日分のログを送信(運用上やりやすい方を選択)

#### メール件名
```
[Tsuricast] 葵ちゃんコメント生成ログ 2026-04-27
```

#### メール本文の構造

```
=== サマリー ===
日付: 2026-04-27
コメント生成数: 137件
キャッシュヒット: 218件
API呼び出し回数: 137回
推定コスト: 53.2円(キャッシュ書込12回・読出218回・新規生成137件)

エラー件数: 2件
モード分布: good=78 / unsure=24 / ng=31 / danger=4

=== 詳細ログ(時系列) ===
[06:00:23] chigasaki-kaigan / 今日 / mode:good
  入力: 月4月、快晴、最低11.4℃、最高18.6℃、波0.6m、北北東3.7m/s(斜め追い風)、
        波周期6.4s、海水温18.6℃、若潮(月齢9.6)、降水量0/0/0/0、sand_beach、正面:S
  出力: 「茅ヶ崎、これは行く日です。波も足元安定で、斜め追い風で投げやすい...」(85字)
  トークン: 入力355(キャッシュ読出16500) / 出力107
  コスト: 0.398円

[06:01:15] shichirigahama / 今日 / mode:good
  入力: ...
  出力: ...
  ...

(すべての生成について同様に記録)

=== エラー(あれば) ===
[14:23:11] tomiura-old-port / 明日: API呼び出しタイムアウト(リトライ1回後失敗)
[18:45:09] hayama-ko / 今日: レート制限ヒット(リトライ後成功)
```

#### 必須項目

各生成について以下を記録:
- タイムスタンプ
- スポットID(またはスポット名)
- 日付ラベル(今日/明日)
- 判定モード(good/unsure/ng/danger)
- **入力値全部**(月、気温、波、風、降水量、潮汐、施設区分、釣り場の正面など、葵ちゃんに渡したデータ)
- **生成コメント全文**
- 入力トークン数(キャッシュ書込/読出/新規入力に分けて)
- 出力トークン数
- このリクエストのコスト(円換算)

#### 実装上の注意

- ログをサーバーローカルにも保存しておく(メール送信失敗時の保険)
- ログファイルは日付ごとに分ける(例: `logs/aoi_2026-04-27.log`)
- 30日以上前のログは自動削除(ストレージ節約)

### 6.3 コストアラート(メール送信)

#### 1日のAPIコスト警告
1日の合計コストが **300円**を超えたら、その時点で警告メールを送信:
```
件名: [Tsuricast 警告] 葵ちゃんコメント1日コストが300円を超えました
本文: 本日の累計コスト: 312円(現在14時時点)、想定の3倍ペースです。
      アクセス急増またはキャッシュ機能の不具合の可能性があります。
```

#### 月間予算警告
月の累計コストが **5,000円**を超えたら警告メール送信。

#### API呼び出し回数異常
1時間あたりのAPI呼び出しが **50回**を超えたら警告メール送信。
通常の想定は1時間あたり数回〜10回程度なので、これを大きく超えると異常。

すべて宛先は `admin@tsuricast.jp`(環境変数 `AOI_NOTIFICATION_EMAIL`)。

## 8. 非機能要件

### 8.1 パフォーマンス

- キャッシュヒット時のレスポンス: **100ms以内**
- キャッシュミス(API呼び出し)時のレスポンス: **5秒以内**(Anthropic APIの応答時間に依存)

### 8.2 同時実行制御

同一スポットの同一日付に対して、複数のリクエストが同時に来た場合、API呼び出しは1回だけにする(他のリクエストは生成完了を待つ)。

実装方式:
- ロック機構(Redisの `SETNX` or データベースの行ロック)
- 5秒以内にロックが取れなければ、コメントなしで返す

### 8.3 ログ

各リクエストに対して以下のログを残す:
- アクセス時刻
- スポットID
- キャッシュヒット/ミス
- API呼び出し有無
- 生成されたコメントのモード(good/unsure/ng/danger)
- API呼び出しコスト(トークン数から計算)

### 8.4 セキュリティ

- ANTHROPIC_API_KEY は環境変数で管理し、コードにハードコードしない
- ログにAPIキーや個人情報を含めない

## 9. テスト要件

### 9.1 単体テスト

- 風向き判定ロジック(`get_wind_relative`)
- 潮汐活発度判定ロジック
- モード判定ロジック(`detect_mode`)
- weather_data_hash 計算

### 9.2 統合テスト

- アクセス → キャッシュなし → API呼び出し → キャッシュ保存 → コメント表示
- アクセス → キャッシュあり → API呼び出しなし → コメント表示
- bot User-Agent → API呼び出しなし → コメントなし(またはプレースホルダー)
- API呼び出し失敗 → エラーハンドリング → コメントなし

### 9.3 負荷テスト(任意)

ピーク日の想定アクセス(194コメント生成 / 1日)を1時間に圧縮した負荷でも安定動作するか確認する。

## 11. 注意事項・制約

### 11.1 法的配慮

葵ちゃんコメントは予報業務法に抵触しないよう設計されている。以下を必ず守ること:

- 葵ちゃんコメントを「予報」として表示しない(タイトルやラベルに「予報」と書かない)
- ページに「※AIによる個人的な感想です。気象予報ではありません」のような旨を控えめに表記
- 数値計算した体感気温などをそのまま表示しない(プロンプト側で対応済みだが、別途UI側で気温表示する場合は注意)

### 11.2 既存の運用への影響

- 既存の16スポット限定テストは段階的に終了する(新方式に統一)
- スポットマスターの spot_type が未設定のスポットがあれば、デフォルトで `sand_beach` を仮定する(または非表示)
- スポットマスターに「釣り禁止」フラグが必要(既存になければ追加実装)。葵ちゃんコメント表示の最優先判定条件として使う

### 11.3 段階的ロールアウトの推奨

650スポット全部に同時展開せず、以下の段階で進めることを推奨:

1. **第1週**: 既存の16スポットのみ実装、安定性確認
2. **第2週**: 神奈川県内スポット約100に拡大
3. **第3週**: 全スポット展開
4. **第4週**: コストとログ確認、不具合修正

各週、コスト実績を確認して想定範囲内か確認する。
